What Was Changed

Made site into a headless CMS.

Added a page specific to each exhibition.

Worked on form validation and sent data to wordpress.

Had an input for first name and last name but chose to combine them into one input for full name.

Went through CSS and HTML to remove any unused/unnecessary code.

Changed hamburger menu.

Made the museum fees section more visible.

Changed wrapper around exhibition cards from DIV to Article tags as well as adding datetime.

How I Changed It

Created the wordpress backend and added all the posts that were in the HTML to the wordpress backend so I could serve them dynamically.

I created each page and fetched the data from the wordpress backend.

I made slight adjustments to how the validation messages were shown and shared the form data with wordpress backend.

Created one input and changed the validation for that input using javascript.

Went through all files and removed code as well as adjusting the tab space from 4 to 2.

I created the hamburger menu from a checkbox hack to creating one using javascript and HTML.

I had this section on the contact page before. Now I added buttons on the landing pages sections near the exhibition cards so that users could be re-directed to that section on another page which provided more information on fees, food, accessibility etc.

As I was getting the information served dynamically I did not have to go through each card item, I could just add the article tag into the javascript which was creating the HTML.

Why I Did It That Way

I decided to create a headless CMS as if this was a real museum they would constantly be updating their exhibitions and news, so this way they could update and delete posts without out knowing any code.

This was done so that users could get more information on each exhibition to help them decide if they would want to go to this specific one or not. Before this was created there was only exhibition cards which users complained that it was not supplying enough information.

I chose to keep the javascript validation instead of just using HTML validation as I felt like you could provide a better user experience with errors etc.

I changed it to one input after reading a couple articles on form experience for the user. It also allowed the form to be shorter for the user to fill in, thus resulting in a better experience.

This was done to reduce the amount of code as well as making it easier for other people who will look at the code in the future.

I did it this way as it looked better and created a smoother user experience when using the nav.

I got feedback from users saying that it was hard to find out if it was free or how much it was to pay. So I made buttons on sections where the users started wondering about that so that the information could be provided to them.

This was done as it would be more semantic serving it within an article tag compared to a DIV. I also added the datetime so that it was more readable by machines and people could easily add it to their calender.
